## Recommended next: Journal Intelligence UI

Why this and not something else:

1. **Data is already accumulating.** Every entry since launch generates `user_pattern_insights` rows. Every day waited is wasted compounding. By now there's likely real signal.
2. **It's the differentiating "magic" feature.** Kiaros's promise over generic astrology apps is "we observe your real patterns over time." That promise is invisible until users see the synthesis.
3. **It teaches you what users actually do.** Once it's clear which patterns land, you'll know how to frame HD/GK signals when Phase 4.5 starts.

### Where to start

Scope before writing code:

1. **Read `lib/ai/blueprint-system-prompt.ts`** — `journalPatternsToText` already formats `user_pattern_insights` for the blueprint prompt. Same shape can drive the UI.
2. **Read migration `0012_journal_intelligence.sql`** — see the `user_pattern_insights` columns (`pattern_type`, `pattern_key`, `sample_size`, `confidence`, `first_seen`, `last_seen`, `summary`, `evidence`).
3. **Sketch the dashboard surface.** Likely lives at `/journal/insights` or as a panel on the Journal page. Each pattern card: pattern name (e.g. "Full Moon in Taurus", "Mercury retrograde", "Mars square Sun"), sample size, confidence, sparkline / list of evidence entries with dates.
4. **Decide on Recharts now or later.** A simple list-view ships in 2 days; Recharts visualization adds 3–4 days. Recommendation: start list-view, add charts in a follow-up.

### Definition of Done

A user with ≥10 journal entries can land on a page and see: (a) which lunar / aspect / retrograde patterns recur in their journaling, (b) how confident the system is, (c) which entries each pattern is built from. RLS holds.
